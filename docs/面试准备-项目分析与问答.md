## 一、项目技术分析

### 1.1 项目全貌

| 维度 | 详情 |
|------|------|
| **项目类型** | 企业级 RAG 智能问答系统 |
| **业务场景** | 中文法律文档检索与问答 |
| **文档规模** | 5 部法律法典，总计约 25 万字（刑法 11 万 + 民法 7.4 万 + 民诉法 3.5 万 + 食品安全法 3 万 + 劳动合同法 1.2 万） |
| **技术栈** | Python 3.12 / FAISS / BM25 / sentence-transformers / LangChain / FastAPI / Gradio / SQLite |
| **LLM** | DeepSeek-V3 (主) / GPT-4o / DeepSeek-R1-Distill-Qwen-7B / 本地 Ollama |
| **Embedding** | BAAI/bge-small-zh-v1.5 (512d, 中文优化) |
| **Reranker** | BAAI/bge-reranker-v2-m3 (Cross-Encoder) |
| **代码规模** | 25+ Python 模块, ~5000+ 行核心代码 |

### 1.2 核心架构

```
用户输入 (Gradio/FastAPI)
    │
    ▼
Query 分类器 (规则引擎: keyword/semantic/default)
    │
    ▼
Embedding 缓存 (LRU 512 + TTL 10min)
    │
    ▼
┌──────────────────────────────────────────┐
│       混合检索 (Hybrid Search)            │
│  ┌──────────────┐  ┌──────────────┐      │
│  │  FAISS 语义   │  │  BM25 关键词  │      │
│  │  (Dense 70%) │  │  (Sparse 30%)│      │
│  └──────┬───────┘  └──────┬───────┘      │
│         └────────┬─────────┘              │
│                  ▼                        │
│         加权合并 (α 动态)                  │
│                  ▼                        │
│     Cross-Encoder 重排序 (Top-5)          │
└──────────────────────────────────────────┘
    │
    ▼
LLM 判断 → 信息不足? → Query Rewrite → 重新检索 (最多3轮)
    │
    ▼
上下文构建 + 矛盾检测 + Prompt 构建
    │
    ▼
LLM 生成 (DeepSeek/OpenAI/Ollama 流式)
    │
    ▼
Metrics 写入 SQLite → 返回用户
```

### 1.3 技术亮点清单

1. **自适应 FAISS 索引引擎**: 根据数据量自动选择 FlatL2 (<1万) / IVFFlat (1万-10万) / IVFPQ (>10万)，通过 IndexIDMap 实现增量增删
2. **Query 分类器**: 基于正则的规则引擎，识别 关键词型 / 语义型 / 混合型 查询，动态调整 Dense/BM25 权重 ($\alpha\in\{0.2,0.7,0.8\}$)
3. **Parent-Child 分块**: 超长文档 (>5万字) 使用大块(parent 2000字)做上下文、小块(child 400字)做检索，平衡检索精度与上下文完整性
4. **双层缓存**: Embedding 查询缓存 (LRU 512 + TTL 10min) + 检索结果缓存 (LRU 128 + TTL 5min)
5. **三后端 LLM 自动检测**: DeepSeek > OpenAI > SiliconFlow > Ollama 优先级链，启动时自动探测可用后端
6. **结构化可观测性**: SQLite 存储全链路 Metrics (每请求 ~20 维度的性能指标)
7. **5 组对照实验 Benchmark**: 分块策略/Embedding模型/检索方式/Query Rewrite/Reranker，自动生成 CSV + 雷达图 + 热力图 + 柱状图
8. **Layout-aware 文档解析**: PyMuPDF blocks + 表格检测 + EasyOCR 内嵌图片识别

## 二、项目包装建议

### 2.1 包装定位

将项目从"个人学习项目"包装为**"面向中文法律领域的企业级 RAG 知识库平台"**。

### 2.2 故事线

> **背景**: 某法律科技公司需要构建一个面向中文法律从业者的智能问答系统，用户可以用自然语言提问法律问题，系统从海量法律文档中检索相关条文并生成准确回答。
>
> **核心挑战**:
> 1. 中文法律文本的语义鸿沟——用户口语 vs 法律术语
> 2. 跨法律混合检索中的内容污染（问刑法返回民法典）
> 3. 超长文档（单部法典 10 万+ 字）的检索精度与上下文完整性
> 4. 多 LLM 后端的灵活调度与成本控制
>
> **我的方案**: 设计并实现完整的 RAG 检索增强生成管线，包括 混合检索、动态Query分类、递归Query改写、交叉编码器重排序、多LLM后端适配 等模块，并通过 Benchmark 体系量化验证各环节优化效果。

|------|------|----------|

## 四、面试问题与回答思路

### ━━━ 模块 A: RAG 基础与架构 ━━━

#### Q1. 你的 RAG 系统整体架构是怎样的？从用户输入到输出回答经过了哪些环节？

**面试官意图**: 考察是否真正理解 RAG 全链路，而非仅仅调包。

**回答思路**:

1. **画架构图** (口述): 用户输入 → Query 分类 → Embedding 编码 (缓存检查) → 混合检索 (FAISS + BM25) → Cross-Encoder 重排序 → LLM 充分性判断 → (不足则 Query Rewrite 再检索) → 上下文构建 → LLM 生成 → 返回

2. **强调设计决策**:
   - 为什么用混合检索而非纯语义？→ 中文法律术语需要 BM25 精确匹配来兜底
   - 为什么加 Query 分类器？→ 不同类型问题适合不同检索权重
   - 为什么两阶段检索 (Recall + Rerank)？→ 工业界标准范式，平衡效率与精度

3. **说出量化结果**: Context Precision 0.76, Context Recall 0.35

**追问**: 如果让你重新设计，你会改变什么？
- 考虑加入多模态支持（法律文书中常有表格/流程图）
- 考虑引入 Graph RAG 处理法律条文间的引用关系（如"依照本法第X条"）

---

#### Q2. 为什么选择 FAISS 而不是直接用 Milvus/Qdrant？什么场景会切换？

**面试官意图**: 考察向量数据库的选型思考，不是只会用默认方案。

**回答思路**:

1. **选 FAISS 的原因**: 轻量零依赖，适合本地部署的 RAG 系统；支持 IndexIDMap 增量操作；自适应索引类型 (FlatL2/IVFFlat/IVFPQ) 覆盖不同规模

2. **FAISS 的局限**: 无分布式能力；无原生过滤查询 (需自己维护 metadata)；持久化需手动序列化

3. **切换 Milvus/Qdrant 的时机**: 多用户并发时 FAISS 无连接管理；需要复杂过滤查询 (按日期/来源筛选)；需要分布式部署

4. **项目中的设计**: 已通过 VectorStore 抽象层预留 Qdrant/Milvus 接口 (`core/vectorstores/` 下有 faiss_store.py, milvus_store.py, qdrant_store.py)，通过 factory 模式切换

---

#### Q3. 混合检索中 Dense 和 BM25 的权重怎么定？$\alpha=0.7$ 是怎么来的？

**面试官意图**: 考察是否有方法论支撑，不是随便设的参数。

**回答思路**:

1. **基础值 0.7**: 参考业界实践 (LlamaIndex 默认 $\alpha=0.7$)，中文法律场景更偏向语义理解

2. **动态调整**: 不是固定 0.7！通过 Query 分类器动态调整：
   - 关键词型查询 (错误码/法条编号如"第二百六十四条"): $\alpha=0.2$ (BM25 优先)
   - 语义型查询 (解释性问题如"偷东西会判刑吗"): $\alpha=0.8$ (Dense 优先)
   - 混合型/默认: $\alpha=0.7$

3. **可验证性**: Benchmark 中有专门的检索方式对比组 (G3)，可量化比较 dense-only/hybrid/BM25-only 的效果

### ━━━ 模块 B: Embedding 与语义匹配 ━━━

#### Q4. 为什么从 all-MiniLM-L6-v2 换成 bge-small-zh-v1.5？实际效果差异有多大？

**面试官意图**: 考察是否理解 Embedding 模型的领域适配问题。

**回答思路**:

1. **问题根因**: all-MiniLM-L6-v2 是英文优化的，将中文法律文本映射到英文语义空间，导致"盗窃"和"租赁"的向量距离很近——这是跨法律污染的根源

2. **选择 bge-small-zh-v1.5 的考虑**:
   - 中文优化 (BAAI 发布，在中文检索评测中表现优异)
   - 512 维 vs 384 维，更丰富的语义表达
   - small 版本平衡了效果与资源 (内存 ~200MB)
   - 后续可升级到 bge-base (768d) 或 bge-m3 (1024d)

3. **Benchmark 中的量化对比**: G2 组设计了 small/base/m3 三个模型的对照实验

**追问**: 如果有更多资源，你会选择什么 Embedding 模型？
- bge-m3: 多语言 + 多粒度，1024d，支持 Dense + Sparse 混合表征
- 或考虑 fine-tune 自己的法律领域 Embedding 模型 (用 SimCSE 或 contrastive learning)

---

#### Q5. Cross-Encoder Reranker 和 Bi-Encoder 的区别？为什么你的系统两者都用？

**面试官意图**: 考察对两阶段检索范式的理解。

**回答思路**:

1. **Bi-Encoder (Embedding 模型)**: Query 和 Document 独立编码 → 向量内积/余弦距离 → 速度快，可以预先计算文档向量 (FAISS 索引)，适合第一阶段的召回 (Recall)

2. **Cross-Encoder (Reranker)**: Query 和 Document 拼接后一起送入模型 → 输出相关性分数 → 更精确但更慢 (不能预计算)，适合第二阶段的精排 (Rerank)

3. **为什么两者都用**: 
   - Recall 阶段: FAISS 从万级 chunks 中召回 10 个候选 (速度快)
   - Rerank 阶段: Cross-Encoder 对 10 个候选精排，输出 Top-5 (精度高)
   - 这是工业界的标准两阶段范式 (Recall + Rerank)

4. **性能数据**: rerank 对比 no-rerank 的 Precision 差异在 G5 Benchmark 中量化对比

### ━━━ 模块 C: Query 改写与上下文工程 ━━━

#### Q6. 递归检索中的 Query Rewrite 是怎么实现的？什么时候停下来？

**面试官意图**: 考察 Agent 式检索的设计思路。

**回答思路**:

1. **触发条件**: 检索完成后，用 LLM 判断当前上下文是否足以回答问题

2. **改写策略**: LLM 基于原始问题和当前检索结果摘要，生成更精准的查询词。例如:
   - "偷了50元洗发水" → "盗窃罪 数额较大 认定标准"
   - "网上发分裂言论" → "煽动分裂国家罪 构成要件"

3. **停止条件**:
   - LLM 判断"信息已足够，不需要进一步查询"
   - 达到最大迭代轮数 (MAX_RETRIEVAL_ITERATIONS=3)
   - 生成内容超长 (>100 字符，视为无效查询)
   - 检索结果为空

4. **Benchmark 验证**: G4 组对比了 raw/legalization/HyDE 三种策略

**追问**: 你有没有遇到过 Query Rewrite 越改写越差的情况？
- 是的，这是常见问题。LLM 可能改写后偏离原意
- 缓解措施: 将原始问题始终带着，改写查询作为补充；限制改写长度；保留原始检索结果

---

#### Q7. 什么是 Parent-Child 分块策略？解决什么问题？

**面试官意图**: 考察对 Chunking 策略的理解。

**回答思路**:

1. **问题场景**: 法律文档通常很长 (单部法典 10 万+ 字)。如果 chunk 太小 (400 字)，LLM 看不到完整的法律条文上下文；如果 chunk 太大 (2000 字)，检索精度下降

2. **Parent-Child 方案**:
   - Parent (大块 2000 字): 保留完整语义上下文，给 LLM 阅读
   - Child (小块 400 字): 做 embedding 索引，用于精确检索
   - 检索用 Child 匹配，返回时带上对应的 Parent 内容

3. **触发条件**: 文本 > 5 万字时自动启用，短文本直接用普通切分

4. **效果**: 平衡了检索精度与上下文完整性，Token 利用率提升 (每个 chunk 携带更多有效信息)

### ━━━ 模块 D: 工程化与性能优化 ━━━

#### Q8. 你的缓存系统是怎么设计的？缓存什么？为什么？

**面试官意图**: 考察系统性能优化思维。

**回答思路**:

1. **两级缓存**:
   - Embedding 缓存: QueryCache (LRU 512, TTL 10min) — 高频查询的 embedding 计算是纯 CPU 耗时 (~50ms)，缓存可以完全跳过
   - 检索结果缓存: QueryCache (LRU 128, TTL 5min) — 完整检索链路 (embedding + BM25 + rerank) 耗时更长，缓存收益更大

2. **为什么用 LRU+TTL 而非 Redis?**
   - 本地单机部署场景，内存缓存足够且更快
   - 法律文档场景数据变更频率低，TTL 5-10min 合理
   - 避免引入额外基础设施依赖

3. **缓存失效策略**: 文档增删时自动清空检索缓存；embedding 缓存有 TTL 兜底

4. **Key 设计**: 归一化处理 (去标点/空格/大小写)，避免"过热怎么办？"和"过热怎么办"被当作不同查询

**追问**: 如果你的系统要支持 1000 QPS，缓存方案需要怎么改？
- Embedding 缓存可加上 Redis 共享缓存，减少多实例重复计算
- 检索结果缓存需考虑一致性 (多用户增删文档时)
- 考虑语义缓存 (Semantic Cache): 语义相似的查询共享缓存

#### Q9. 你的 Metrics/日志系统是怎么设计的？怎么用这些数据做优化？

**面试官意图**: 考察可观测性思维和数据驱动优化能力。

**回答思路**:

1. **设计**: SQLite 单表 `rag_logs`，记录每请求的 ~20 个维度:
   - 时间维度: 总延迟、检索延迟、重排序延迟、LLM 延迟
   - 质量维度: 检索到的 chunks、rewritten queries
   - 成本维度: prompt tokens、completion tokens、total tokens
   - 命中维度: cache_hit、query_type、hybrid_alpha
   - 业务维度: session_id、model、web_search、error

2. **线程安全**: threading.Lock 保证并发写入

3. **数据分析价值**:
   - 缓存命中率 → 决定是否需要扩大缓存容量
   - P99 检索延迟 → 是否需要优化索引或切换向量库
   - Query 分类分布 → 调整 $\alpha$ 默认值
   - Token 消耗趋势 → 成本预估

#### Q10. 为什么要支持多个 LLM 后端？怎么实现自动切换？

**面试官意图**: 考察多模型调度和容灾设计。

**回答思路**:

1. **为什么多后端**:
   - 成本优化: DeepSeek (便宜) vs OpenAI (效果好) vs Ollama (免费本地)
   - 容灾: 一个 API 挂了自动切另一个
   - 场景适配: 简单查询用便宜模型，复杂推理用强模型

2. **自动检测优先级**: DeepSeek → OpenAI → SiliconFlow → Ollama
   - 通过检查 .env 中的 API Key 是否存在且不为占位符
   - 回退到本地 Ollama 时检测 localhost:11434 是否可达

3. **统一调用接口**: OpenAI 兼容格式，所有云端 API 共用 `_call_cloud_api()` 函数

**追问**: 如果让你实现一个 LLM 网关/Router，你会怎么做？
- 基于成本 + 延迟 + 可用性的多目标路由
- 小问题→便宜的模型 (如 DeepSeek-V3)
- 复杂推理 → 强模型 (如 GPT-4o)
- 实现 fallback 链: A 超时 → B → C

### ━━━ 模块 E: Benchmark 与评估 ━━━

#### Q11. 你的 Benchmark 系统是怎么设计的？5组对照实验分别验证什么？

**面试官意图**: 考察实验设计能力和科学思维。

**回答思路**:

1. **设计原则**: 控制变量法 — 每组只改变一个变量，其他参数保持 baseline 不变

2. **5 组实验**:
   - G1 分块: fixed(400) vs recursive vs article(法条级) — 验证分块策略对语义完整性的影响
   - G2 Embedding: bge-small(512d) vs bge-base(768d) vs bge-m3(1024d) — 验证模型规模与维度对中文法律文本理解的影响
   - G3 检索: Dense vs BM25 vs Hybrid — 验证混合检索的必要性
   - G4 Rewrite: raw vs legalization vs HyDE — 验证 Query Rewrite 对 Recall 的提升
   - G5 Rerank: no-rerank vs bge-reranker — 验证重排序对 Precision 的提升

3. **评估指标**: RAGAS 四维度 — Faithfulness (忠实度), Context Precision (上下文精确度), Context Recall (上下文召回率), Answer Relevancy (回答相关性)

4. **实验基础设施**: 缓存复用 (chunked_texts, built_indices) 避免重复计算；支持 `--group G1` 单独运行

**追问**: 你觉得 RAGAS 评估靠谱吗？有什么局限性？
- RAGAS 也依赖 LLM 打分，存在 LLM 偏差 (LLM-as-judge bias)
- 法律领域的 Faithfulness 评估尤其难——需要法律专业知识
- 补充: 考虑加入人工评估 + 检索命中率 (hit@k) 等更客观的指标

---

#### Q12. 你们的 Baseline 数据中 Context Recall 只有 0.35，你觉得主要瓶颈在哪？怎么改进？

**面试官意图**: 考察问题诊断能力和优化思路。

**回答思路**:

1. **主要瓶颈 (按影响力排序)**:
   - **Query-Document 语义鸿沟**: 用户口语 vs 法律条文术语，Embedding 模型难以跨越
   - **Chunk 语义不完整**: 固定大小切分导致法律条文被截断，丢失上下文
   - **跨法律污染**: 5 部法律在一个索引，问刑法可能匹配到民法典

2. **改进方案 (与 Benchmark 结论一致)**:
   - P0: Query Rewrite (HyDE/法律化) — 预期 Recall 提升 60-70%
   - P0: 法条级分块 (article-level) — 预期 Recall 提升 50-60%
   - P1: 升级到 bge-m3 (1024d) — 预期全面提升 10-15%
   - P1: 混合检索 + source filtering — 降低跨法律污染

3. **Benchmark 的价值**: 不是猜测，而是通过对照实验量化验证每个改进的实际效果

### ━━━ 模块 F: 系统设计 & 场景题 ━━━

#### Q13. 如果法律文档从 5 部增加到 500 部 (50GB 文本)，你的系统哪些地方会出问题？怎么解决？

**面试官意图**: 考察大规模系统的设计能力。

**回答思路**:

1. **瓶颈分析**:
   - FAISS 内存: 几百万向量时 IndexIVFPQ 需要大量内存，单机扛不住
   - BM25 重建: 每次增删全量重建，万级尚可，十万级以上耗时严重
   - Embedding 编码: 全量 re-index 耗时数小时

2. **解决方案**:
   - 向量存储: 切换到 Milvus 分布式集群 (已预留接口)
   - BM25: 切换到 Elasticsearch 做关键词检索
   - Embedding: 批处理 + GPU 加速 + 增量更新
   - 分库分表: 按法律类型分 collection (刑法/民法/行政法)
   - 检索路由: 先分类问题属于哪个法律领域，再定向检索

#### Q14. 设计一个"AI 法律助手"系统，要求支持多轮对话、法规更新、引用溯源。

**面试官意图**: 考察系统设计完整性和产品思维。

**回答思路**:

### ━━━ 模块 G: 八股延伸 (基础知识) ━━━

#### Q15. RAG 中的 "Lost in the Middle" 问题是什么？你怎么解决的？

**回答**: LLM 对长上下文中部信息关注度低。我的方案: (1) Cross-Encoder 重排序确保最相关 chunk 在前; (2) RERANK_TOP_K=5 限制上下文数量; (3) Parent-Child 模式用大块代替多个小块。

#### Q16. 向量检索中 FlatL2、IVFFlat、IVFPQ 的区别是什么？

**回答**: FlatL2 暴力搜索 O(N)，精确但慢，适合 <1万; IVFFlat 倒排索引+聚类，近似搜索 O(sqrt(N))，适合 1万-10万; IVFPQ 乘积量化，压缩向量，适合 >10万但精度有损。我的 AutoFaissIndex 根据数据量自动选型。

#### Q17. BM25 的两个超参数 k1 和 b 分别控制什么？

**回答**: k1 控制词频饱和度(默认 1.2)，b 控制文档长度归一化(默认 0.75)。项目中用 rank_bm25 默认值，因为主要做混合检索的补充信号而非独立使用。

#### Q18. Cross-Encoder 和 Bi-Encoder 的时间复杂度对比？

**回答**: Bi-Encoder 是 O(Q+D) 编码 + O(Q×D) 点积 (可预计算 D 向量); Cross-Encoder 是 O(Q×D) 次联合推理，每次推理成本远高于点积。所以 Bi-Encoder 做召回，Cross-Encoder 只在少量候选上精排。

## 五、面试技巧总结

### 5.1 关键话术

| 场景 | 话术 |
|------|------|
| 被问为什么选某技术 | "我做了对照实验验证" (引出 Benchmark 系统) |
| 被问效果如何 | "我们通过 RAGAS 四维评估量化对比" (说出具体数字) |
| 被问遇到什么问题 | "中文法律领域的主要挑战是语义鸿沟，我们通过 Query Rewrite + 混合检索解决" |
| 被问为什么不做 X | "Benchmark 数据显示 X 的收益优先级较低，我们优先做了 Y" |
| 被问如何优化 | "通过 Metrics 系统分析 P99 延迟/缓存命中率分布，定位瓶颈" |

### 5.2 项目讨论时的加分项

1. **展示 Benchmark 图表**: 雷达图/热力图直观展示各方案对比
2. **展示 Metrics 数据**: 证明系统在生产环境运行过
3. **展示代码结构**: 模块化设计体现工程能力
4. **主动提 trade-off**: 如"IVFPQ 比 FlatL2 快但精度略低，我们根据数据量自动选择"

### 5.3 避免的回答方式

- ❌ "我用 LangChain 的 RAG 模块直接搭的" — 显得只会调包
- ❌ "效果挺好的" — 没有量化数据
- ❌ "就是按照教程做的" — 没有自己的思考
- ✅ "我设计并实现了一套完整的 RAG 管线，包括 X/Y/Z，并通过 Benchmark 验证了..."
